# 核心价值说明

## 设计原则

### 1. 编辑器无关
- ❌ 不依赖特定 IDE（VSCode、Cursor、JetBrains 等）
- ❌ 不依赖特定 AI 模型（Claude、GPT、Gemini 等）
- ✅ 使用通用的 Markdown 格式
- ✅ 使用标准的文件路径约定

### 2. AI 驱动判断
- ❌ 不使用硬编码的代码解析规则
- ❌ 不依赖语言特定的 AST 解析
- ✅ 让 AI 理解变更的语义含义
- ✅ Subagent 基于自然语言规则做决策

### 3. 代码共存
- ❌ 不将模块文档集中存放在单独目录
- ✅ 模块文档与代码目录共存
- ✅ 文档跟随代码移动（重构友好）
- ✅ 提高文档的可发现性

### 4. 职责驱动
- 文档核心：清楚描述每个目录/模块**负责什么**
- 不是需求文档，不是变更日志
- 是项目逻辑结构的**当前状态快照**

### 5. 自动化维护
- 用户只提需求，系统自动维护文档
- 基于 Git Hook 触发，无需人工记忆
- 增量更新，不重新生成

## 核心价值

### 对 AI 编程助手的价值

| 价值点 | 说明 |
|--------|------|
| **快速上手** | AI 在新对话中能在 5 分钟内理解项目结构 |
| **准确定位** | 能正确回答"功能 X 在哪实现？" |
| **正确判断** | 能正确判断"新功能 Y 应该加在哪？" |
| **避免重复** | 了解现有功能，不创建重复实现 |
| **遵循约定** | 理解项目约定，生成符合规范的代码 |

### 对开发团队的价值

| 价值点 | 说明 |
|--------|------|
| **降低上手成本** | 新成员可以通过 AI 快速理解项目 |
| **知识沉淀** | 项目逻辑结构有持久化的文档 |
| **自动维护** | 文档随代码自动更新，不会过时 |
| **统一理解** | 团队对项目结构有一致的认知 |

### 与传统文档的区别

| 维度 | 传统文档 | AI Context 文档 |
|------|----------|-----------------|
| **目标读者** | 人类开发者 | AI 编程助手 |
| **更新方式** | 手动维护 | 自动+手动 |
| **核心内容** | 如何使用 | 职责是什么 |
| **粒度** | 详细说明 | 精炼导航 |
| **维护成本** | 高（易过时） | 低（自动同步） |

## 技术约束

### 开发约束
- **Python 版本**: 3.6+（仅使用标准库）
- **跨平台**: macOS/Linux/Windows
- **性能**: Git hook 执行应在 5 秒内完成

### 文档约束
- **格式**: Markdown
- **大小**: MAP.md < 20KB，_AI_CONTEXT.md < 10KB
- **命名**: `_AI_CONTEXT.md`（下划线前缀，便于排序）

### Git 约束
- **非侵入**: 文档更新失败不应阻断 commit
- **原子性**: 文档更新应与代码变更在同一 commit
- **可选**: 可通过配置禁用自动更新

## 适用场景

### 适用的项目类型
- ✅ 中大型项目（多模块、多目录）
- ✅ 持续开发的项目（非一次性脚本）
- ✅ 多人协作的项目
- ✅ 使用 Git 版本控制的项目

### 不适用的场景
- ❌ 单文件脚本项目
- ❌ 纯配置/数据项目
- ❌ 一次性或临时项目
- ❌ 不使用 Git 的项目
